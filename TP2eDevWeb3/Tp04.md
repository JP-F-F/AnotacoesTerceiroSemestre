## Utilizando o Git Bash - primeiros passos

**Alguns Comandos**:
* Listar documentos: ``dir`` ou ``ls``
* Limpar tela: ``clear``
* Versão git: ``git -version``
* Ver histórico de commits: ``git log`` 
* Ver histórico de commits de forma detalhada: ``git log -p`` 
* Ver histórico de commits de forma personalizada: ``git log --pretty="parametros de formatacao"`` ([link pra esses parametros](https://devhints.io/git-log-format))
* Criar um repo para gestão de versões: ``git init --bare``
* Ver caminho dos diretórios: ``git remote -v``
* Renomear o diretório: ``git remote rename <nome_atual> <nome_novo>``
* Criar nova Branch: ``git branch <nome da nova Branch>``
* Listar Branchs: ``git branch``
* Mudar de Branchs: ``git checkout <nome da branch>``
* Fazer o merge: ``git merge <branch_alvo>``
* Fazer Rebase: ``git rebase <branch_alvo>``
* Visualizar as Modificações de forma gráfica: ``git log --graph``
* Descartar/desfazer uma alteração num arquivo: ``git checkout --<nome_do_arquivo>``
* Descartar/desfazer a adição de um arquivo: ``git reset head <nome_do_arquivo>``
* Descartar/desfazer um commit: ``git revert <hash_do_commit>``
* Salvar alterações de forma temporária: ``git stash``
* Ver o que está no stash: ``git stash list``
* Recuperar e remover a ultima stash: ``git stash pop``
* Buscar a stash: ``git stash <nº_Temporario>``
* Descartar a stash retirada: ``git stash drop``
* Voltar o repositório para um commit: ``git checkout <nº_do_commit>`` 

**Gravando alterações em seu repositório**:

Ao executar o comando git status, recebemos algumas informações que talvez não estejam tão claras principalmente quando nos deparamos com termos como HEAD, working tree, index, etc.

* HEAD: Estado atual do nosso código, ou seja, onde o Git os colocou
* Working tree: Local onde os arquivos realmente estão sendo armazenados e editados
* index: Local onde o Git armazena o que será commitado, ou seja, o local entre a working tree e o repositório Git em si.

**Como funciona o gerenciamento de arquivos com o git**

Todos os arquivos gerenciado pelo Git são armazenados como um blob numa estrutura de pastas localizado em **“/.git/objects”**.
Esses arquivos ficam organizados em commits em uma estrutura com uma _mensagem_ e uma _tree_.

**Ignorando arquivos**:

Existe um arquivo chamado .gitignore no qual você pode listar todos os arquivos e diretórios que deseja que o Git Ignore.

Para saber que arquivos ignorar aqui algumas dicas:
* Arquivos sem relevância para o projeto (anotações pessoais por exemplo);
* Arquivos de configurações do IDE de desenvolvimento;
* Arquivos que não diz respeito ao funcionamento do projeto;

**Quando commitar?**:

**Regra básica é: Só faça um commit com o software funcionando.**
Recomenda-se:
* Quando uma nova funcionalidade for implementada
* Quando um erro, bug ou correção foi realizada
* Quando o projeto foi alterado alguma especificação

## Compartilhamento do Projeto

podemos criar um o “vinculo” através de um caminho ou como o próprio nome diz “um repositório remoto”.
Podemos criar quantos diretórios remotos forem necessários, da seguinte forma:
````
$ git remote
$ git remote add <nome_do_diretorio> <caminho_do_diretorio>
$ git remote add Repositorio_Local c:/Aula_Git/Servidor
````

Neste exemplo, ele busca e atualiza no mesmo caminho. Em casos mais complexos (servidores compartilhados), podemos ter servidores/caminhos diferentes.

**Criando um "clone" do projeto**

Supondo que um usuário, membro da equipe, deseja desenvolver uma aplicação para esse mesmo projeto. Para isso essa pessoa deve trazer para seu diretório o projeto (não alterar o arquivo remoto);

Esse novo membro da equipe, acessando pelo diretório dele deve CLONAR o projeto para seu diretório (normalmente uma maquina local). Para isso utiliza-se o comando:

````
$ git clone <caminho_do_repositório_remoto> <nome_do_dir_novo>
$ git clone /c/Aula_Git/Servidor Projeto_Aula_Git
````
